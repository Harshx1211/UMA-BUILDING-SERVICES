## Local SQLite Schema

The SQLite database (`sitetrack.db`) is created automatically by `lib/database.ts â†’ initializeSchema()` on every app launch.

It **mirrors** the Supabase schema exactly, plus two SQLite-only tables:

| Table | Notes |
|-------|-------|
| `meta` | Key-value store â€” holds `schema_version` |
| `sync_queue` | Offline write queue â€” **never synced to Supabase** |

### SQLite Version History

The app uses a versioned migration system. Version is stored in `meta` (`key = 'schema_version'`).
Each `if (currentVersion < N)` block in `initializeSchema()` runs **exactly once** and is idempotent.

| Version | What Changed |
|---------|-------------|
| **0** | Initial schema â€” all core tables created |
| **1** | `users.push_token TEXT` â€” Expo push notification token |
| **2** | `job_assets.checklist_data TEXT`, `job_assets.is_compliant INTEGER` |
| **3** | `CREATE INDEX idx_job_assets_asset_id` â€” faster previous-result lookups |
| **4** | `sync_queue.retry_count INTEGER`, `sync_queue.last_error TEXT` â€” retry limiting |
| **5** | `assets.variant TEXT`, `assets.asset_ref TEXT` â€” asset sub-type and site reference |

> **Adding a new SQLite migration:**
> 1. Increment `CURRENT_SCHEMA_VERSION` in `lib/database.ts`
> 2. Add a new `if (currentVersion < N) { ... }` block following the existing pattern
> 3. Add the corresponding Supabase `ALTER TABLE` to `docs/supabase_migrations.sql`
> 4. Update the tables above in this file

## Sync Architecture

```
User Action (e.g. Fail an Asset)
  â”‚
  â–¼
Zustand Store (updateAssetResult)
  â”‚  writes immediately
  â–¼
Local SQLite (upsertRecord â†’ job_assets)
  â”‚  queues write
  â–¼

  â”‚
  â–¼ (when online, every 60 s)
Background Sync Engine (lib/sync.ts)
  â”‚  pushes pending items
  â–¼
Supabase (Postgres) â€” REST API
```

**Key rules:**
- UI always reads from SQLite â€” never Supabase directly
- Supabase is append-only from the app's perspective; it never overwrites local state
- `upsertRecord()` uses `ON CONFLICT(id) DO UPDATE SET` â€” never `INSERT OR REPLACE` â€” to preserve FK child rows
- Items that fail 5+ pushes are marked `synced = -1` and abandoned (check `last_error`)

## Key Architecture Decisions

### Signature Storage
Signatures are uploaded to Supabase Storage (`job-photos/signatures/{job_id}/signature.png`).
Only the **public URL** is stored in `signatures.signature_url`. Base64 is never stored in the DB.

### Column Name Security
All generic CRUD helpers (`insertRecord`, `updateRecord`, etc.) validate column names via
`_safeColumnName()` before interpolating into SQL â€” prevents SQL injection via attacker-controlled keys.

### Asset Type System
Asset types are defined in `constants/AssetData.ts` â€” the single source of truth for:
- Official type names (must match `assets.asset_type` values in DB)
- Sub-variants per type
- Icons, colours, inspection routines
- Compliance checklist mapping (`constants/Checklists.ts`)
